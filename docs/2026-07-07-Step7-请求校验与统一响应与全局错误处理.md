# Step 7 · 请求校验 + 统一响应格式 + 全局错误处理

> 目标：把 Step 6 故意留下的三个粗糙处一次填平——
> ① 请求体没校验 ② 响应格式不统一 ③ Prisma 错误直接吐 500 HTML。
> 这三样合起来，就是一个后端「对外契约」是否专业的分水岭。

## 本步做了什么

1. `src/lib/errors.ts`：自定义错误体系 `AppError` + `NotFoundError` / `ConflictError` / `ValidationError`
2. `src/lib/api-response.ts`：统一响应信封 + `sendSuccess()` 助手
3. `src/schemas/user.schema.ts`：用 zod 定义创建/更新的请求体规则
4. `src/middlewares/validate.ts`：校验中间件工厂，进控制器前先校验 `req.body`
5. `src/middlewares/error-handler.ts`：全局错误处理中间件（4 参数签名），所有错误的统一出口
6. 改造 `user.controller.ts`：控制器变薄——throw 错误、走 `sendSuccess`
7. `user.routes.ts` 写操作加 `validate(...)`；`not-found.ts` 改为抛 `NotFoundError`；`app.ts` 末尾挂 `errorHandler`

## 统一响应契约

**成功**：

```json
{ "success": true, "data": { "id": 1, "email": "a@b.com", "name": "A" } }
```

**失败**（状态码由错误类型决定）：

```json
{ "success": false, "error": { "code": "VALIDATION_ERROR", "message": "请求参数校验失败", "details": [{ "field": "email", "message": "email 格式不正确" }] } }
```

前端只需判断 `success`，成功读 `data`、失败读 `error`，全站一致。
（唯一例外：`DELETE` 成功返回 `204 No Content`，按 HTTP 语义不带响应体。）

## 三个核心设计

### 1. 校验前置：控制器不再写防御代码

校验做成**中间件**，挂在控制器前面：

```
请求 → validate(createUserSchema) → postUser 控制器
         │ 校验失败                    │ 校验通过时才会执行
         ▼                            ▼
   next(ValidationError)          req.body 已是干净、带类型的数据
```

`validate` 通过后会用 `schema.safeParse` 的结果**覆盖 `req.body`**——多余字段被剥掉、类型已转换。
所以控制器里再也没有 `if (!email)` 这种防御代码，`req.body as CreateUserBody` 直接可信。

> zod schema 还用 `z.infer` 反推出 TS 类型（`CreateUserBody`），
> 校验规则和类型定义**同一份来源**，改规则时类型自动跟着变。

### 2. 错误分层：Service 抛语义错误，中间件翻译成 HTTP

Service 不认识 HTTP，它只会说「这个用户不存在」——于是 `throw new NotFoundError(...)`。
到底返回什么状态码，由全局错误处理中间件统一决定。三种错误来源都汇聚到这里：

| 来源                     | 例子                       | 翻译结果          |
| ------------------------ | -------------------------- | ----------------- |
| 我们抛的 `AppError`      | `NotFoundError` / 校验失败 | 用它自带的状态码  |
| Prisma `P2002`（唯一冲突）| 重复 email                 | `409 CONFLICT`    |
| Prisma `P2025`（记录不存在）| 更新/删除不存在的 id       | `404 NOT_FOUND`   |
| 其它未知错误             | 代码 bug                   | `500`（并打日志） |

> 关键：**5xx 才打日志**（意料之外，需排查），4xx 是客户端问题不刷屏；
> 生产环境不返回 `details`，避免泄露内部信息。

### 3. 为什么错误处理中间件能「兜住一切」

它有个特殊签名——**4 个参数** `(err, req, res, next)`。Express 正是靠参数个数认出它，
所以必须挂在 `app.ts` 的**最后**。请求的错误通过三种方式流到它：

- 控制器 `throw`（Express 5 会自动捕获 async 函数里的 reject 并转发）
- 校验中间件 `next(new ValidationError(...))`
- Service 里 Prisma 抛出的错误一路 reject 上来

对比 Step 6：那时没有它，`PATCH` 一个不存在的 id 会掉进 Express 默认处理器 → 一整页 500 HTML。
现在同样的请求，返回的是规范的 `404 + JSON 信封`。

## 动手验证

```bash
docker compose up -d && pnpm dev
B=http://localhost:3000/api/users

# ✅ 创建成功 → 201 + success 信封
curl -X POST $B -H 'Content-Type: application/json' -d '{"email":"a@b.com","name":"A"}'

# ❌ 邮箱格式非法 → 400 VALIDATION_ERROR（details 指出是哪个字段）
curl -X POST $B -H 'Content-Type: application/json' -d '{"email":"bad"}'

# ❌ 邮箱重复 → 409 CONFLICT（Prisma P2002 被翻译）
curl -X POST $B -H 'Content-Type: application/json' -d '{"email":"a@b.com"}'

# ❌ id 非数字 → 400；查不到 → 404；PATCH 空 body → 400；改不存在的 id → 404
curl $B/abc
curl $B/99999
curl -X PATCH $B/99999 -H 'Content-Type: application/json' -d '{}'
curl -X PATCH $B/99999 -H 'Content-Type: application/json' -d '{"name":"X"}'
```

以上每一条返回的都是统一的错误信封 + 恰当的状态码——这就是 Step 6 那页 500 HTML 的终点。

## 小结

到这一步，users 这套接口已经有了**专业后端该有的对外契约**：进来的数据一定被校验，
出去的响应格式统一，各种异常都被优雅地转成语义化的状态码。这套错误/响应基础设施是
**全局通用的**，之后新增任何资源都能直接复用。
