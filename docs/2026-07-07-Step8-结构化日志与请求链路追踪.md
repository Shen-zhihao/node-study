# Step 8 · 结构化日志（pino）+ 请求链路追踪

> 目标：把散落各处的 `console.log` 升级成生产级日志——结构化（JSON）、分级、高性能，
> 并用 `AsyncLocalStorage` 给每个请求一个 `requestId`，让它自动贯穿所有层的日志。

## 本步做了什么

1. 安装 `pino`（日志库）+ `pino-pretty`（开发环境美化，devDependency）
2. `src/config/env.ts`：新增 `LOG_LEVEL` 配置（校验 + 默认 info）
3. `src/lib/request-context.ts`：用 `AsyncLocalStorage` 实现请求上下文，存 `requestId`
4. `src/lib/logger.ts`：pino 实例，`mixin` 自动把当前 `requestId` 注入每条日志
5. `src/middlewares/request-logger.ts`：重写——生成/沿用 requestId、开上下文、记结构化访问日志
6. 把 `server.ts` / `error-handler.ts` 里的 `console.*` 换成 `logger`；service 层加一条示例日志

## 为什么不用 console.log

| 维度   | `console.log`          | `pino`                                       |
| ------ | ---------------------- | -------------------------------------------- |
| 格式   | 纯文本，机器难解析     | 结构化 JSON，可被 ELK/Loki 检索、告警        |
| 分级   | 无                     | trace/debug/info/warn/error/fatal，可过滤    |
| 性能   | 同步写，拖慢主流程     | 低开销，序列化走旁路                         |
| 上下文 | 要自己拼字符串         | 字段化 + mixin 自动附加公共字段              |

开发环境经 `pino-pretty` 转成彩色单行文本，肉眼友好；生产环境输出原始 JSON 交给采集系统。
两种形态由 `NODE_ENV` 自动切换。

## 请求链路追踪：一个 id 串起一次请求的所有日志

**痛点**：一个请求穿过 中间件 → 控制器 → service，想让每层日志都带同一个 `requestId`，
难道要给每个函数都加一个 `requestId` 参数、层层往下传？侵入性太强。

**解法**：`AsyncLocalStorage`（Node 内置）——一块「跟着异步调用链自动流动」的存储。

```
请求进来
  │
  ▼ request-logger 中间件：
  │    ① 生成/沿用 requestId
  │    ② res.setHeader('x-request-id', id)   ← 回写响应头，前端也能拿到
  │    ③ runWithContext({ requestId }, next) ← 开启上下文，包裹后续整条链路
  ▼
控制器 → service → ...   （这条链路上任意深度）
  │
  ▼ 任何一层调用 logger.xxx()
       └─ logger 的 mixin 自动 getRequestId() 并注入这条日志
```

关键三块拼图：

1. **`request-context.ts`**：`runWithContext(store, cb)` 起上下文，`getRequestId()` 随时取回——不传参。
2. **`logger.ts` 的 `mixin`**：每条日志输出前调用，返回 `{ requestId }` 并入日志。
3. **`request-logger.ts`**：在请求入口把 `next` 包进 `runWithContext`。

### 实测：同一 requestId 贯穿多层

给创建请求带上 `x-request-id: trace-demo-123`，日志里 **service 层的「创建用户」** 和
**中间件的「请求完成」** 用的是同一个 id：

```
INFO: 创建用户          requestId: "trace-demo-123"   email: "dave@example.com"   ← service 层
INFO: 请求完成          requestId: "trace-demo-123"   method: POST  url: /api/users  statusCode: 201  durationMs: 67   ← 中间件
```

没带 header 的请求会自动分配一个 UUID。生产环境里，靠这个 id 就能把一次请求散落在各层的
日志全部串起来——这就是「链路追踪」。

> 小贴士：`x-request-id` 优先沿用上游（网关/前端）传来的值，所以多个服务之间也能用同一个 id
> 贯穿，实现**跨服务**的链路追踪。

## 一个细节：为什么 env.ts 还留着 console

`logger` 依赖 `env`（要读 `LOG_LEVEL`），所以 `env.ts` 自己不能反过来用 `logger`——
那是「鸡生蛋」。而且环境变量校验失败属于**启动前的致命错误**，此时日志系统还没建好，
用 `console.error` + `process.exit(1)` 直接快速失败反而最清晰。这是刻意保留的例外。

> 另外你会看到 `prisma:query ...` 的 SQL 日志——那是 Prisma 自己的日志（Step 5 配的
> `log: ['query',...]`），和 pino 是两套，不受 LOG_LEVEL 影响。

## 动手验证

```bash
docker compose up -d && pnpm dev

# 观察响应头里的 x-request-id
curl -i http://localhost:3000/api/health

# 带上自定义 id，看它如何贯穿 service 日志和访问日志
curl -X POST http://localhost:3000/api/users \
  -H 'Content-Type: application/json' \
  -H 'x-request-id: my-trace-001' \
  -d '{"email":"a@b.com","name":"A"}'

# 想看更详细的调试日志？在 .env 里设 LOG_LEVEL=debug 再启动
```

## 小结

日志从「给人看的字符串」升级成了「给机器处理的结构化数据」，并且每条日志都能定位到
是哪一次请求产生的。这套日志 + 上下文基础设施是全局的，之后所有代码打日志都自动受益。
